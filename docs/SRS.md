## 1. 引言

### 1.1 目的
本文件定义"穿搭分身"App 在 **MVP 阶段** 的软件需求，作为产品、设计、研发、测试的共同基线与验收依据。

### 1.2 项目背景与范围
用户上传穿搭照片后，App 提供 **AI 图文穿搭点评**；并可生成一张 **风格化"角色设定卡"**（虚拟形象 + 服装拆解图），把卡中的服装 **拆解成单品** 存入 **数字衣柜**，用户再从衣柜 **拖动单品给一个风格化虚拟形象换装**。

本产品在 MVP 阶段 **明确选择"风格化虚拟形象"路线（A 方向）**：虚拟形象是被美化的数字分身，**不追求与用户真实体型一致**。

### 1.3 术语定义
| 术语 | 含义 |
|------|------|
| 角色卡 / 拆解卡 | 由图像生成模型一次性生成的版式图，包含虚拟形象多视图 + 服装平铺拆解 + 配色等（参见产品参考图） |
| 生成（Generation） | 模型输出**新像素**，可将一身衣服重画为白底、互不遮挡的平铺图 |
| 分割 / 切件（Segmentation） | 把一张已有图片切成**带透明背景的独立单品**（每件一张 PNG + 掩膜 + 包围盒） |
| 纸娃娃换装（Paper-doll） | 在 2D 固定姿势形象上，叠加可拖拽的服装 PNG 图层完成换装 |
| 上身重生成 | 选定若干单品后，把【形象 + 单品】交给图像模型**重新生成一张穿好的图**，由模型负责版型与遮挡 |
| 身份保持生成 | 用一张自拍反复生成"同一个人"的风格化形象（保人脸神似，不保身材） |
| Provider | 被抽象封装的第三方/自托管 AI 能力（点评 / 生成 / 切件） |

### 1.4 参考与约定
- 优先级采用 MoSCoW：**Must / Should / Could / Won't（本期不做）**。
- 需求编号：功能需求 `FR-n`，非功能需求 `NFR-x`，接口 `IF-x`。

## 2. 总体描述

### 2.1 产品愿景与定位
做一款"会点评、能拆衣、可换装"的 **个人穿搭娱乐 + 灵感工具**：以"生成你的风格化穿搭分身"为情绪钩子，以"衣柜 + 换装"为留存玩法。

### 2.2 功能概览
```
① 上传穿搭（本人穿的 / 网上效果图） → ② AI 图文点评
                                        └→（按需）③ 生成角色卡式拆解图
                                                     └→ ④ 服装切件入「衣柜」
                                                          └→ ⑤ 2D 换装（纸娃娃）
                                                               └→（可选）⑥ 上身效果重生成
        ⑦ 虚拟形象创建/管理（自拍身份保持生成 或 选预设）
```

### 2.3 用户角色与特征
| 角色 | 特征 | 主要诉求 |
|------|------|----------|
| 分享型用户 | 活跃于社交平台、爱出片 | 生成好看的角色卡用于分享 |
| 整理型用户 | 衣服多、纠结搭配 | 攒衣柜、找搭配灵感与点评 |
| 娱乐型用户 | 喜欢捏人/换装小游戏 | 玩虚拟分身换装 |

### 2.4 运行环境
- **客户端**：iOS App（SwiftUI），iPhone。
- **服务端**：自有后端（API + 对象存储 + 数据库）。
- **AI 能力**：以 Provider 形式接入第三方或自托管模型（见 §5）。
- **网络**：在线（核心生成/切件依赖云端算力）。

### 2.5 设计与实现约束
- **C-1** 仅 iOS（本期不做 Android / Web）。
- **C-2** 角色卡与形象生成 **必须在云端**完成（端上无同级算力），故 **用户人像照片需上传至服务端/第三方接口**（见 §4 隐私）。
- **C-3** **本期不做 3D 形象、不做 3D 换装、不做 AR 全身试穿**。
- **C-4** 虚拟形象为风格化形象，**不承诺与真实体型一致**。
- **C-5** AI 能力须做 Provider 抽象，便于更换模型或自托管，避免厂商锁定。
- **C-6** 须满足上架地区的个人信息保护法规（如 PIPL / GDPR）。

### 2.6 假设与依赖
- **A-1** 依赖第三方 AI 接口（多模态 LLM、图像生成、分割）的可用性、配额与稳定价格。
- **A-2** 图像生成对"颜色/廓形/品类"可较好保真，但 **不保证复刻具体单品细节**（logo、版型、五金）。
- **A-3** 用户设备具备相机/相册访问与基本网络。

## 3. 功能需求

> 每条含：优先级、描述、输入/输出、验收标准（AC）。

### FR-1 穿搭图片上传 — **Must**
- 描述：用户可从相册/相机上传一张穿搭图，支持"本人穿着图"或"网上效果图"两类来源标注。
- 输入：图片（JPEG/PNG/HEIC）。输出：受控的图片资源 + 元数据。
- AC：
  - 支持常见格式与 ≤N MB 限制，超限有提示与压缩。
  - 上传前展示 **隐私同意说明**（含人像将上传云端处理，见 FR-11）。

### FR-2 AI 图文穿搭点评 — **Must**
- 描述：基于上传图，返回 **结构化文字点评**（如整体印象、配色、版型、场合、改进建议）+ 可选灵感图。
- 输入：图片（+ 可选用户偏好/场合）。输出：点评文本（分维度）+ 可选图。
- AC：
  - 点评分维度返回，至少包含"亮点/建议"两类。
  - 失败可重试；不阻塞其他功能。
  - 首屏文本反馈应有流式或加载态（见 NFR-1）。

### FR-3 角色卡式拆解图生成 — **Must**
- 描述：按需根据上传穿搭，生成一张 **风格化角色卡**（虚拟形象 + 服装平铺拆解 + 配色），风格与原穿搭的"组成/样式/颜色"保持一致。
- 输入：穿搭图（+ 形象设定）。输出：角色卡图 + 结构化元信息（单品清单、配色）。
- AC：
  - 生成的服装在 **组成、廓形、主色** 上与原图一致（人工抽检通过率目标见 NFR）。
  - 生成过程提供进度态；失败可重试/降级。

### FR-4 服装拆解切件入库 — **Must**
- 描述：把角色卡中的服装平铺区 **切成独立单品**（透明 PNG + 类别标签 + 包围盒），保存进衣柜。
- 输入：角色卡图（白底平铺区）。输出：N 个单品对象。
- AC：
  - 每个单品为透明背景 PNG，含 `label / bbox / 主色 / 来源图引用`。
  - 同类多件或成对单品（如左右鞋）尽量拆为独立项；不可拆时允许手动修正。
  - 提供基础人工修边/重命名/删除入口。

### FR-5 数字衣柜管理 — **Must**
- 描述：浏览、筛选、增删用户的单品；单品来源含"切件结果"。
- AC：按类别（上衣/下装/鞋/配饰…）筛选；支持删除与重命名；空态有引导。

### FR-6 虚拟形象创建与管理 — **Must**
- 描述：用户拥有一个风格化虚拟形象，用于换装。形象来源二选一：
  - (a) **自拍身份保持生成**：上传自拍 → 生成"神似本人"的风格化形象（保脸、不保身材）；
  - (b) **预设/可微调形象**：从内置风格化形象中选择。
- AC：
  - MVP **至少实现 (b)**；(a) 作为增强项接入身份保持能力。
  - 形象生成后 **固定保存并跨会话复用**（避免身份漂移）。
  - 一名用户可保存 ≥1 个形象。

### FR-7 2D 纸娃娃换装 — **Must**
- 描述：用户把衣柜单品拖到 **固定姿势的虚拟形象** 上，自动吸附到锚点（肩/腰/脚），处理图层叠放，组成一套搭配。
- 输入：形象 + 选中的单品集合。输出：一套 2D 搭配（图层组合）。
- AC：
  - 拖拽/缩放/移除流畅（见 NFR-1 帧率）。
  - 同部位互斥替换（换上衣替换原上衣）；图层顺序正确（外套盖内搭）。
  - 可保存为搭配。

### FR-8 上身效果重生成 — **Should**
- 描述：对选定搭配，调用图像生成 **重画一张"形象穿好整套"的图**，由模型负责版型/褶皱/遮挡，弥补纯图层叠加的生硬。
- AC：输入【形象 + 单品集合】，输出一张合成上身图；有进度态；失败回退到 FR-7 的图层效果。

### FR-9 搭配保存 / 收藏 / 分享 — **Should（分享为 Could）**
- AC：可保存搭配并命名；可导出/分享角色卡或上身图（分享前提示内容与隐私）。

### FR-10 账户与数据 — **Should**
- AC：支持登录（含 Sign in with Apple）；用户可查看/导出/删除自己的数据。

### FR-11 隐私同意与数据删除 — **Must**
- 描述：首次使用人像相关功能前取得明确同意；提供一键删除人像/形象数据。
- AC：
  - 明示"哪些数据上传、上传给谁、保留多久"（见 NFR-3）。
  - 默认 **不长期留存原始人像图**（处理后即弃，除非用户选择保存）。
  - 提供删除账户与全部数据的入口。

### 本期不做（Won't / Out of Scope）
- AR 全身试穿；3D 形象与 3D 换装；面向真实体型的精确试穿（B 方向）；电商购买/比价；社区/社交流。

## 4. 非功能需求

### NFR-1 性能与体验
- 点评文本首屏反馈 ≤ ~10s（建议流式）。
- 角色卡生成 P50 ≤ ~30s、切件 ≤ ~5s，全程显式进度态。
- 换装画布交互 ≈ 60fps，拖拽无明显卡顿。

### NFR-2 可靠性
- 所有云端调用具备超时、重试与降级路径；单个 AI 步骤失败不致整流程不可用。

### NFR-3 隐私 / 安全 / 合规（关键）
- 满足上架地区个人信息保护法规（PIPL / GDPR 等）。
- 人像数据 **最小化收集、明示同意、用完即弃**；优先选择 **承诺不留存、不用于训练** 的 Provider。
- 传输 TLS、静态存储加密；越权访问防护；未成年人保护。

### NFR-4 内容安全（信任与安全）
- 对 **用户上传** 与 **模型生成** 的图像进行内容审核（拒绝违法/未成年人/明显违规内容），并遵守各 Provider 使用政策。

### NFR-5 成本可控
- 每次生成/切件可计量与归因；提供配额、缓存与降级策略以控制单位成本。

### NFR-6 可维护 / 可扩展
- AI 能力以统一 Provider 接口封装，模型可热替换 / 在线/自托管可切换。

### NFR-7 兼容性
- 目标 iOS 版本 ≥ 16（若使用 Vision 前景分割等能力，相应功能要求 ≥ 17）。

## 5. 外部接口需求

### IF-1 用户界面
- iOS 原生（SwiftUI）：上传、点评、角色卡、衣柜、换装画布、形象管理、设置/隐私。

### IF-2 AI 能力接口（Provider 抽象）
| 能力 | 用途 | 候选实现 |
|------|------|----------|
| 多模态 LLM（视觉） | 穿搭图文点评 | Claude（视觉）/ GPT-4o / Gemini |
| 图像生成 / 编辑 | 角色卡、形象、上身重生成 | GPT-4o 图像 / Gemini（Nano Banana）/ Flux + InstantID·IP-Adapter·PuLID（身份保持） |
| 分割 / 抠图 | 切件（白底平铺图，易场景） | BiRefNet + 连通域（托管于 fal.ai / Replicate，或自托管） |

### IF-3 后端 API
- HTTPS / REST；鉴权（Token）；对象存储（图片）；数据库（元数据）。

|------|----------------|
